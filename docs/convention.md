# Convention

## 작업 방식

- 큰 구조 변경 전에는 팀원에게 먼저 공유한다.
- 기능 구현이 우선이므로 과한 추상화와 리팩터링은 미룬다.
- 공통 로직이나 공용 컴포넌트 수정은 사용 범위를 한번 확인하고 반영한다.

## Git 전략

- 기본 브랜치는 `main`, 개발 브랜치는 `dev`로 사용한다.
- 작업은 각자 `feature/*` 브랜치에서 진행한다.
- 작은 수정은 빠르게 공유하고 머지한다.
- 공용 컴포넌트, 상태, 구조 변경은 PR로 공유한 뒤 반영한다.
- PR이나 공유 메시지는 변경 이유와 확인 방법만 짧게 남긴다.

## 네이밍 규칙

- 변수명, 함수명: `camelCase`
- 컴포넌트명: `PascalCase`
- 타입, 인터페이스명: `PascalCase`
- 폴더명: `kebab-case`
- 상수명: `UPPER_SNAKE_CASE`

## 이미지 규칙

- 이미지는 `public/images` 아래에 저장한다.
- 파일명은 `kebab-case`로 작성한다.
- 일반 이미지는 `webp` 사용을 우선한다.
- 아이콘, 로고, 단순 도형은 `svg`를 사용한다.
